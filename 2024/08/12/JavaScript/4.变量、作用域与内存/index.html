4.变量、作用域与内存

kitty2024-08-122025-04-06

原始值与引用值

原始值：

  • 最简单的数据
  • Undefined,null,boolean,number,string,symbol 六种 一般放在栈中
  • 保存原始值的变量是按值访问，操作的是存储在变量中的实际值

引用值：

  • 多个值构成的，保存在内存中的对象
  • 操作对象时，实际上操作的是该对象的引用而非实际对象本身
  • 保存引用值的变量是按引用访问的

复制值

原始值上 复制值与被复制 两个变量都是独立的

引用值上 复制的值是一个指针，指向存储的堆内存中的对象，实际上两个变量指的是同一个对象

传递参数

所有的函数参数都是按值传递的，意味着函数外的值会被复制到函数内部的函数中

表明函数中参数的值改变后，原始的引用仍然没变

当 obj 函数内部被重写时，它变成一个指向本地对象的指针，而那个本地对象在函数执行结束时已被摧毁

垃圾产生

JS 的引用数据类型是存在堆中，然后在栈内存中保存一个堆内存中实际对象的引用

栈内存中保存了一个地址，这个地址和堆内存中实际值是相关的

垃圾产生的原因是因为引用数据类型发生了重新赋值的过程中，失去了与上一个对象的引用关系，一多就爆内存

垃圾回收

JS 是使用垃圾回收的语言，是执行环境负责在代码执行时管理内存。通过自动内存管理实现内存分配和闲置资源回收

离开作用域的值会被自动标记为可回收，然后在垃圾回收期间被删除

基本思路

确定哪个变量不会再使用，然后释放它占用的内存。整个过程是周期性的，隔一段时间就自动运行

函数中的局部变量会在函数执行时存在，栈（或 堆）内存会分配空间以保存相应的值。函数在内部使用了变量，然后退出。此时，就不再需要那个局部变量了，它占用的内存可以释放，供后面使用

垃圾回收程序如何判断哪些变量不会再使用？

标记清理

变量进入上下文(作用域)，比如在函数内部声明一个变量，该变量会被加上存在于上下文中的标记，当变量离开上下文时，会被加上离开的标记

标记过程的实现并不重要，关键是策略

  • 垃圾回收程序运行时，会标记内存中所有变量，默认全是垃圾
  • 将所有在上下文中的变量、被在上下文中变量引用的变量标记去掉，把不是垃圾的改成 1，在此之后再被加上标记的就是待删除的，
  • 因为任何上下文中的变量都访问不到它们(此时已经离开上下文)了，随后垃圾回收程序做一次内存清理销毁带标记的所有值并收回内存

优点：

实现简单 用二进制就可以标记

缺点：

  • 内存碎片：清除后，剩余的对象内存位置不变会导致空闲内存空间不连续，出现内存碎片，能会出现分配所需内存过大的对象时找不到合适的块
  • 分配速度慢：操作仍是一个 O(n)，大对象的分配效率会更慢

如何找到合适的块？

  • First-fit，找到大于等于 size 的块立即返回(推荐)
  • Best-fit，遍历整个空闲列表，返回大于等于 size 的最小分块
  • Worst-fit，遍历整个空闲列表，找到最大的分块，然后切成两部分，一部分 size 大小，并将该部分返回(不推荐)

缺点补充-标记整理

标记结束后，标记整理算法会将不需要清理的对象向内存的一端移动，最后清理掉边界的内存

引用计数

对每个值都记录被引用的次数，被引用+1 被覆盖-1 为 0 代表无法再访问到，此时可以收回内存

产生的问题：

  • 循环引用：对象 A 有一个指针指向对象 B，而对象 B 也引用了对象 A
  • JavaScript 引擎不再使用这种算法，但某些旧版本的 IE 仍然会受这种算法的影响，原因是 JavaScript 会访问非原生 JavaScript 对象 如 DOM 元素 设置为 null 才能切断关系
  • 但是 ie9 就把 DOM 和ＯＭ给换成 JS 对象 ， 这同时也避免了由于存在两套垃圾回收算法而导致的问题，还消除了常见的内存泄漏现象

优点

引用计数在引用为 0 时，就立刻被回收

缺点

需要个计数器，无法解决循环引用无法回收的问题

对垃圾回收的优化

分代式垃圾回收机制

新老生代

新生代的对象为存活时间短的对象=>新产生的对象一般只 1-8M 的容量

老生代为存活时间较长或常驻内存的对象，经历过新生代垃圾回收后存活下来的对象，容量通常比较大

两个生代垃圾回收器不同

新生代垃圾回收

新加入的对象会存放到使用区，当使用区快写满时，需要执行一次垃圾清理

如何进行垃圾操作？

  • 新生代垃圾回收器对使用区中的活动对象做标记
  • 标记成功后将使用区的活动对象复制到空闲区+排序
  • 清理阶段-将非活动对象占用空间清理
  • 将使用区变空闲区，空闲区变使用区

如果一个对象经过多次复制还存活=>生命周期较长对象=>移到老生代

对象太大影响后期内存分配=>移到老生代

并行回收

全停顿：JS 是一门单线程语言，是运行在主线程上的，在进行垃圾回收时会堵塞 JS 脚本的执行。需要等待垃圾回收完后再恢复运行

如果一次垃圾回收需要 20ms 那应用逻辑就得停 20ms，时间过长可能会造成页面卡顿的问题

并行回收：叫上两个辅助线程和主线程一起干活，但是主线程还是需要让出来的，只是时间变短了，这个过程内存是静态的，不需要考虑内存中对象的引用关系改变，只需要考虑协同

新生代对象空间就采用并行策略

在执行垃圾回收的过程中，会启动了多个线程来负责新生代中的垃圾清理操作

这些线程同时将对象空间中的数据移动到空闲区域，这个过程中由于数据地址会发生改变

所以还需要同步更新引用这些对象的指针，此即并行回收

老生代垃圾回收

用的标记清除算法，因为老生代中的对象通常比较大，用新生代的处理方式耗时

从一组根元素开始，递归遍历这组根元素，遍历过程中能到达的元素=>活动对象，没达到=>非活动对象，直接清非活动对象，V8 会使用标记整理算法来优化空间

主要是使用并发标记

标记完成后，再执行并行清理

清理任务会采用增量方式分批在各个 JS 任务之间执行

为什么需要分代式？

把一些新、小、存活时间短的对象作为新生代，采用一小块内存频率较高的快速清理

而一些大、老、存活时间长的对象作为老生代，使其很少接受检查

新老生代回收机制及频率不同，此机制出现很大程度提高了垃圾回收机制的效率

增量标记与懒性清理

并行策略虽然好，但它还是全停顿式的，对于老生代来说如果对象太大，那还是会消耗大量的时间，所以Ｖ８将全停顿标记切换成增量标记

增量

将一次 GC 标记的过程分成多个小步，每执行完一小步就停一会，让应用逻辑执行一会，交替着来完成一轮 GC 标记

每小次 GC 标记完后如何暂停下去执行任务，又如何恢复的，如果在暂停标记的后，将标记好的对象引用关系修改了又咋办？

三色标记法
背景：

老生代采用的标记清理算法，在没有增量算法之前，单纯使用黑色和白色来标记

垃圾回收器将所有的数据置白色->垃圾回收器将所有访问到的数据标为黑色->标记为黑色的数据对象是活动对象，剩余白色是带清理的垃圾对象

但是这种标记法，在垃圾回收器执行了一段增量回收后，暂停后启动主线程执行了应用程序中的一段 JS 代码，垃圾回收器再次被启动，这时内存中黑白色都有，无法得知下一步走到哪里

  • 白色指的是未被标记的对象
  • 灰色指自身被标记，成员变量（该对象的引用对象）未被标记
  • 黑色指自身和成员变量皆被标记
  • 从白->灰->黑

有灰色标记，恢复时直接从灰色阶段开始继续执行

无灰色标记直接进入清理阶段

优点：

不需每次都扫描整个内存空间。可以很好的配合增量回收进行暂停恢复的操作，减少全停顿时间

写屏障
背景

在增量中修改引用，比如ＡＢ C 都要清理了，突然有了新引用，但是 B 要被回收了，后面的程序还要用到 D，就不对了

一旦有黑色对象引用白色对象，该机制会强制将引用的白色对象改为灰色，保证下一次增量 GC 标记阶段可以正确标记＝＞强三色不变性即 D 强制改为灰色

惰性清理

对活动对象和非活动对象做标记，对真正的清理释放内存 V ８采用的惰性清理

  • 增量标记完后，假如当前的可用内存足以让我们执行代码，那是不需要立刻清理内存
  • 将清理程序延迟一下，也无需一次性清理完所有的非活动对象内存，可以按需逐一进行清理直到所有的非活动对象内存清理完毕，后面再执行增量标记
增量标记与惰性清理优缺
优
  • 主线程停顿时间大大减少
  • 让用于与浏览器交互过程变流畅
缺
  • 没有减少主线程总暂停时间甚至稍微增加
  • 写屏障机制成本
  • 增量标记可能会降低应用程序的吞吐量
并发回收

因为并行回收会阻塞主线程，增量标记同样增加了总暂停时间

在主线程执行 JS 过程中，辅助线程能够在后台完成执行垃圾回收的操作，辅助线程在执行垃圾回收时主线程可以自由执行不被挂起

缺点

需要考虑主线程在执行 JS 时，堆中对象引用关系随时都有可能变化，这时辅助线程之前做的标记或者正在进行的标记也会改变，所以需要实现一些读写锁来控制

性能

  • 以前调度垃圾回收程序的策略

是根据分配数比如 256 个变量或 64kb 字符串，只要满足某个条件，垃圾回收程序就会运行但是问题在于，其整个生命周期内始终需要那么多变量，结果就会导致垃圾回收程序过于频繁地运行

  • 现在改为动态改变分配变量、字面量或数组槽位等会触发垃圾回收的阈值

如果垃圾回收程序回收的内存不到已分配的 15%， 这些变量、字面量或数组槽位的阈值就会翻倍。如果有一次回收的内存达到已分配的 85%，则阈值重置为默认值

内存管理

JS 的运行是在一个内存管理和垃圾回收都特殊的环境，浏览器的内存通常分配的比较少，所以为了避免运行大量 JS 的网页耗尽系统内存而导致操作系统崩溃，需要将内存占用量保持在一个较小的值

1.保证只保存必要的数据

接触引用：如果数据不再必要，设置为 null，释放其引用，局部变量在超出作用域后会被自动解除引用

适合全局变量和全局对象的属性

解除对一个值引用不会自动导致相关内存被回收

2.通过 const 和 let 声明提升性能

因为都以块为作用域，相比起 var，使用这两个新关键字可能会更早地让垃圾回收程序介入，尽早回收应该回收的内存

当不再用到的对象内存没有及时被回收

全局变量

在没有任何关键字声明的变量，解释器会把变量 name 当作 window 的属性来创建，这样 window 不被清理就不会消失，所以需要加 let const 关键字

定时器的回调通过闭包引用了外部变量

游离 DOM 引用

我们代码中进行 DOM 时会使用变量缓存 DOM 节点的引用

移除节点的时候，我们应该同步释放缓存的引用，否则游离的子树无法释放

遗忘的定时器

只有当定时器结束后 执行clearInterval ,clearTimeout才会被回收,没有被 clear 会造成内存泄漏

遗忘的事件监听器

当事件监听器在组件内挂载相关的事件处理函数

而在组件销毁时不主动将其清除

其中引用的变量或函数都被认为是需要的而不会被回收

遗忘的监听者模式

遗忘的 Map,Set 对象

使用 Map 或 Set 存储对象,同 Obj 都是强引用

JS 垃圾回收机制是如果我们持有一个对象的引用,那么这个对象就不会被垃圾回收,指的是强引用

如果需要使用Set 或者 Map,可以采用weakSet 或者 weakMap,只允许存储对象弱引用的唯一值,不会干扰 JS 的垃圾回收

弱引用指的是一个对象若只被弱引用所引用，则被认为是不可访问的，因此可能在任何时刻被回收

可以在控制台看到数据输出,是因为浏览器保存了我们输出对象的信息数据引用

5.静态分配与对象池

压榨浏览器减少浏览器执行垃圾回收的次数，合理使用分配的内存

  • 不要过度动态创建矢量对象

对象更替的速度是运行垃圾回收程序的一个标准

resultant 生命周期很短，很快失去所有对它的引用，成为可以被回收的值

如果被频繁的调用那么回收程序会发现对象更替的速度很快，从而更频繁地安排回收

如果需要在其他地方实例化矢量 resultant ，addVector的行为没有变可以考虑用对象池进行创建

对象池
  • 在初始化时，创建一个对象池用来管理一组可回收的对象
  • 应用程序可以向该对象池请求一个对象设置属性使用
  • 操作完成后再还给对象池
  • 因为没发生对象初始化，垃圾回收检测不会发现有对象更替

如果对象池是在对象不存在时创建新的，在对象存在时复用，那么实现的本质是有单调增长但为静态的内存 ，需要使用某种结构维护所有对象用数组比较好

静态分配是优化的一种极端形式

内存三大件

内存泄漏

对象不再使用但是没有被回收,内存没有被释放

内存膨胀

短时间内内存占用极速上升到达一个峰值,想要避免需要使用技术手段减少堆内存的占用

频繁 GC

GC 执行的特别频繁，一般出现在频繁使用大的临时变量导致新生代空间被装满的速度极快，而每次新生代装满时就会触发 GC,想要避免的话就是不要弄太多的临时变量,不用了就会被回收
