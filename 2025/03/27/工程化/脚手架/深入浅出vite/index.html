深入浅出vite

了解前端工程化：https://vue3.chengpeiquan.com/engineering.html#csr-%E4%B8%8E-ssr

一.初识

痛点&解决

  • 模块化需求

由于业内模块标准很多(ESM,CommonJS,AMD,CMD…)，所以需要使模块正常加载+兼容不同模块规范适应不同执行环境

- 解决：提供模块加载方案，并兼容不同的模块规范
  • 兼容浏览器、编译高级语法

受限浏览器的实现规范，高级语法/语言(TS,JSX…)要在浏览器正常运行，需要转化成浏览器可以理解的，这需要工具链层面的支持

- 解决：配合Sass\Tsc\Babel等前端工具链完成高级语法的转译功能，
同时对静态资源也能进行处理，使作为一个模块正常加载
  • 线上代码质量

在生产环境中，不仅考虑代码安全性、兼容性、性能问题，保证线上代码正常运行，由于浏览器版本较多，代码兼容和安全策略各不同

- 解决：在生产环境中，配合Terser等压缩工具进行代码压缩和混淆，
通过Tree Shaking删除未使用的代码，提供对低版本浏览器的语法降级处理等
  • 开发效率

项目的冷启动/二次启动时间、热更新时间在项目越来越大的时候会影响开发效率

- 解决：构建工具本身通过各种方式来进行性能优化，
包括使用原生语言go/rust、no-bundle等，提高项目的启动性能和热更新的速度

为什么Vite是当前最高效的构建工具

对比起Webpack，一般使用Webpack之后，启动时间、热更新时间、代码打包过长

  • 项目冷启动时必须递归打包整个项目的依赖树
  • JS语言本身的性能限制，导致构建性能遇到瓶颈影响开发效率

vite很好地解决了这些问题

  • Vite 底层使用的构建引擎是Esbuild 和 Rollup
  • 开发效率=>Vite 在开发阶段基于浏览器原生 ESM 的支持实现了no-bundle服务，另一方面借助 Esbuild 超快的编译速度来做第三方库构建和 TS/JSX 语法编译
  • 模块化=>Vite 基于浏览器原生 ESM 的支持实现模块加载，并无论是开发环境还是生产环境，都可将其他格式的产物(如 CommonJS)转换为 ESM
  • 语法转译=>Vite 内置了对 TypeScript、JSX、Sass等高级语法的支持，也能够加载各种各样的静态资源，如图片、Worker 等等。
  • 产物质量=>Vite 基于成熟的打包工具 Rollup 实现生产环境打包，同时可以配合Terser、Babel等工具链，可以极大程度保证构建产物的质量

二.前端模块化演进

无模块化标准阶段

1.文件划分-最原始

将应用的状态和逻辑分开到不同的文件中，通过script引入

风险

  • 模块中的变量相当于全局声明和定义，会有变量名冲突问题
  • 变量都在全局定义，很难知道某个变量到底属于哪个模块，给调试带来困难
  • 无法清晰管理模块之间的依赖关系和加载顺序，例如逻辑执行顺序只能通过手动调整
2.命名空间

解决文件划分方式中全局变量定义所带来的一系列问题

3.IIFE(立即执行函数)

模块作用域的区分更加彻底

每个IIFE都会创建一个私有的作用域，在私用作用域中的变量外界是无法访问的，只有模块内部的方法才能访问

模块私有成员功能：data变量只能在模块内部的method函数中通过闭包访问

缺点

但是无论是IIFE或者是文件划分方式都无法真正解决–模块加载

如果模块间存在依赖关系，那么 script 标签的加载顺序就需要受到严格的控制，一旦顺序不对，则很有可能产生运行时 Bug

CommonJS规范

业界最早正式提出的 JavaScript 模块规范，主要用于服务端

定义了一套完整的模块化代码规范，Node.js为之实现了自动加载模块的loader

  • 统一的模块化代码规范
  • 实现自动加载模块的加载器(也称loader)

require导入，module.exports导出模块

问题

  • 模块加载器由Node.js提供，依赖Node.js本身的功能实现，但是将CommonJS模块直接放在浏览器中无法执行，当然现在也有社区第三方的loader
  • CommonJS约定以_同步_模块加载，这种加载机制在服务端没问题。一模块在本地不需要网络IO，二只有服务启动时才会加载模块，而服务通常启动后会一直运行所以对服务的性能并没有太大影响
  • CommonJS放在浏览器端会带来性能问题，会产生大量的同步模块请求，浏览器需要等待响应返回后才能继续解析模块。模块请求会造成浏览器JS解析过程的阻塞，导致页面加载速度缓慢

总结：是一个不太适合在浏览器中运行的模块规范

AMD规范

全称：Asynchronous Module Definition 异步模块定义规范

不会像CommonJS规范进行同步加载

这样模块的代码执行之前浏览器会先加载依赖模块

也可以用require

require和define的区别：前者只能加载模块而不能定义一个模块

问题

规范使用稍复杂

同期出现的规范当中也有 CMD 规范，这个规范是由淘宝出品的SeaJS实现的，解决的问题和 AMD 一样。不过随着社区的不断发展，SeaJS 已经被requireJS兼容了。

可能也听说过 UMD (Universal Module Definition)规范，它并不算一个新的规范，是兼容 AMD 和 CommonJS 的一个模块化方案，可以同时运行在浏览器和 Node.js 环境

ES6 Module

官方提出的规范，得到了现代浏览器的内置支持，Node.js中也得到了原生支持，具由跨平台能力，它也是社区各种生态库的发展趋势，尤其是被如今大火的构建工具 Vite 所深度应用

在 HTML 中加入含有type=”module”属性的 script 标签，那么浏览器会按照 ES Module 规范来进行依赖加载和模块解析

开发模式下
no-bundle

  • 快速启动和热重载（HMR）：在浏览器中直接加载单独的文件（每个 JavaScript 文件、CSS 文件等会被单独请求），而不会将它们合并成一个大文件。这减少了构建时间，使得开发过程更高效
  • 模块按需加载：根据页面加载时需要的模块动态地去请求，而不是一开始就将所有文件打包成一个捆绑文件。这样每次修改文件后，只有相关的部分会重新加载，而不是整个应用

生产模式下要捆绑？

  • 性能优化：捆绑和代码分割可以减小加载的资源大小。例如，如果你有多个页面或者组件，Vite 会根据需要将代码分割成多个块，并在用户请求时按需加载，这样可以加快页面加载速度。
  • 减少请求次数：如果所有的 JavaScript 和 CSS 文件都被捆绑成一个或者几个文件，浏览器只需要进行少量的请求。能显著提高页面加载速度。
  • 优化静态资源：把所有的资源（JS、CSS、图片等）打包，并执行压缩，移除不必要的代码，使得最终的生产包体积更小，加载速度更快。

这也是 Vite 在开发阶段实现 no-bundle(无捆绑) 的原因，由于模块加载的任务交给了浏览器，即使不打包也可以顺利运行模块代码
